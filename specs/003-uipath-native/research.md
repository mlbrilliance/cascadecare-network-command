## R1. Package entry-points install failure (GATING) — RESOLVED (root cause found)

**Decision**: Each install-failing package needs a valid `entry-points.json`
(schema `https://cloud.uipath.com/draft/2024-12/entry-point`) declaring its entry
point(s) with input/output schemas. The API-workflow packages (ProjectType `Api`)
and the BPMN package ship without one, so Orchestrator's install-time entry-points
extraction fails (API → Error 2005 "Entry points configuration missing/corrupted";
BPMN → Error 1654 "entry points definition invalid").

**Evidence**:
- Installed-OK coded agent `agents/claim-flow-anomaly-detector/`: `project.uiproj`
  → `ProjectType: "Function"` + a complete `entry-points.json` with `entryPoints[]`
  (`filePath: "main"`, `uniqueId`, `type: "function"`, full input/output JSON
  schemas, call `graph`). Auto-generated by `uip codedagent init`.
- Failing API workflow `api_workflows/counsel-hawthorne/`: `project.uiproj` →
  `ProjectType: "Api"`, `main: "main.json"`, and **no `entry-points.json`**.
- BPMN start event declares `<uipath:entryPointId value="Entry_IdealIncidentResponse"/>`
  — that entry point must be materialized in the package's entry-points definition.

**Rationale**: Install-time extraction reads `entry-points.json` per package;
pack+publish don't — which is exactly why these "pack+publish fine but won't
install." The fix is per-package authoring, not a manifest/solution change.

**Approach (Phase 2/implement)**: (1) check for a CLI generator for `Api`/Maestro
project types (analog to `codedagent init`'s entry-point emission); (2) else author
`entry-points.json` by hand from each package's `main.json` input/output contract,
mirroring the coded-agent shape minus the Python-specific `graph`; (3) validate one
package's install before fanning out to all 14 + BPMN.

**Alternatives rejected**: drop API/BPMN from deploy (loses the live integrations —
that's the slimmed-core fallback already in place); all-mocks (defeats the
Integration Service story).

## R2. Maestro folder / release-key context — RESOLVED (fallback documented)

**Decision**: Drive R1→R5 dry runs from the **Maestro UI**; treat CLI
`case process run` as best-effort.

**Rationale**: CLI needs a release-key from `case process list -f <folder-key>`,
but that list rejects every `uip or folders list` key, and folder list returns
opaque keys with no names, so `CascadeCare-Core` can't be mapped. The Maestro UI
resolves folder context interactively and is the demo surface judges watch anyway.

**Alternatives**: keep probing the Maestro-specific folder API (deferred — off the
critical path once the UI path works).

## R3. Canvas fan layout coords — RESOLVED

**Decision**: Edit the master caseplan `layout` block so the 6 spawn tasks share
one `y` with evenly-pitched, monotonically increasing `x` (BAA-analysis agent left
of the fan). Pin by round-trip download if Studio Web re-flows on open.

**Rationale**: V20 stores canvas positions in `layout` — the only lever for a
"visible horizontal fan." The download-and-pin pattern already proved necessary
for this project's BPMN variables and SLA fields; reuse it.

**Alternatives rejected**: manual editor drag only (not reproducible on clean
deploy; the slice wants a repeatable hero moment).

## R4. UiPath Apps branding surface — RESOLVED (via Slice 013 research)

**Decision**: Apply logo/palette/typography in the **app.json screen/components
definition** under `apps/clearflow-network-command/` — NOT the Python backend. Per
Slice 013 research, the App is a Coded App: `type: "App"` manifest with a `screens[]`
array and a `components` tree (UiPath Apps Studio Web V2 JSON); the Python backend
(`backend/dashboard.py`) only assembles the `DashboardPayload` data. Branding =
theme/style on the screen + a logo asset reference; the four visual zones
(cascade_tree, reversal_timeline, agent_activity, override_controls) stay as-is.

**Rationale**: Editing the data layer wouldn't change appearance; the screen
definition is the brandable surface. Fictional ClearFlow brand only (IP safety II).

**Alternatives rejected**: defer to Slice 017 (it's an explicit Slice 015 task and
fully offline-unblocked); restyle the backend (wrong layer).
